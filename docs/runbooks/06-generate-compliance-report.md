# Runbook 06 — Generate a Compliance Report

> **Status against RC1 freeze (`dceaabc`):** Describes the
> production operator surface. The `mai-admin compliance report`
> and `mai-admin compliance verify` commands cited below are
> **not declared** in the RC1 `mai-admin` CLI (`Command` enum at
> `tools/mai-admin/src/main.rs:36-50` ships only `backup`,
> `restore`, plus stubbed `audit`, `trust`, `vault`). The
> corresponding HTTP paths on a running daemon are
> `POST /v1/compliance/reports/generate` and
> `GET /v1/compliance/reports/{id}/download` (per
> `mai-api/src/routes.rs:255-263`); re-verification at rest uses
> `verify_certified_report` from `mai-compliance::reports`.

## When to use

- Quarterly compliance attestation.
- Auditor or counsel request.
- After a policy change, to demonstrate the new bundle is in
  force across all observed traffic.
- Post-incident, to bound the affected request set.

## Pre-flight

Run the audit chain verification first
([05-verify-audit-chain](05-verify-audit-chain.md)). A report
generated against an unverified chain has no provenance.

## Steps

1. Decide the window. Pick wall-clock bounds with timezone:
   ```bash
   START="2026-01-01T00:00:00Z"
   END="2026-03-31T23:59:59Z"
   ```
2. Generate the report:
   ```bash
   sudo -u mai mai-admin compliance report \
        --start "$START" \
        --end   "$END" \
        --out   /var/lib/mai/reports/q1-2026.json \
        --sign
   ```
   `--sign` produces a sibling `.sig` file signed by the
   compliance signing key. Unsigned reports are for internal
   review only and are explicitly rejected by counsel-facing
   workflows.
3. Capture the report digest for the operator log:
   ```bash
   sha3sum /var/lib/mai/reports/q1-2026.json
   ```
4. Hand the `.json` + `.sig` pair off via the documented out-of-band
   channel. Do not email; do not push to a shared drive without
   encryption.

## What the report contains

- Time bounds and the WAL `last_seq` covered.
- Per-policy-module: requests evaluated, blocked, redacted,
  allowed; counts only — never payload content.
- Bundle id(s) in effect during the window.
- Trust anchor fingerprints active during the window.
- Audit chain head + checkpoint coverage.
- Generator version, host id, signing key fingerprint.

The report contains no regulated payload content. If the
requesting party needs payload content, that is a separate
review path that exits MAI entirely.

## Verifying a report later

```bash
sudo -u mai mai-admin compliance verify \
     /var/lib/mai/reports/q1-2026.json \
     --signature /var/lib/mai/reports/q1-2026.json.sig \
     --anchors /etc/mai/trust-anchors
```

Exit `0` means the report was generated by this fleet's signing
key against an intact chain at the time of generation. Re-running
later does **not** prove the live chain has not since diverged;
run `audit verify` separately for that.

## Failure modes

| Symptom | Cause | Fix |
|---|---|---|
| `chain_unverified` | `audit verify` was skipped or failed | Run [05-verify-audit-chain](05-verify-audit-chain.md) first |
| `window_overlaps_outage` | Daemon was down for part of the window | Document the outage explicitly in the report metadata; do not silently include partial data |
| `signature_failed` on verify | Report was tampered with, or signing key rotated mid-window | Cross-reference signer fingerprint; consult counsel |

## Do not

- Do not generate a report without `--sign` for anything
  counsel-facing.
- Do not include payload content in the report; the schema
  rejects it.
- Do not regenerate a report and discard the original. Each
  generation is itself an audit event; keep all generations.
